THE CORE IDEA

A skill is a workflow, not a document

  • A skill tells Claude how to do something, step by step Not what good looks like. Not principles. Actual steps with checkpoints and evidence.
  • It lives as a markdown file, invoked with a slash command Type /pr-sweep and Claude knows which tools to call, in what order, and what output to produce.
  • The purpose is execution, not information The default behavior of any AI agent is to take the shortest path to done. Skills make the invisible work mandatory.
ANATOMY

Five parts of a well-built skill

1
Trigger & description The slash command and when to activate. Also: when not to use it.
2
Numbered steps — process over prose Imperative instructions the agent executes in order. "Run gh pr diff" — not "reviewing PRs is important."
3
Output format & destination HTML report? GitHub review with inline comments? Slack message? Specify so output is consistent every time.
4
Exit criteria Concrete evidence the work is done — not "tests pass," but "save test output showing 0 failures."
5
Anti-rationalization table Pre-written rebuttals to excuses the agent hasn't told yet.
●
What are Exit Criteria?
The proof that work is actually done. Not "I think it works" but artifacts you can verify: a test log with 0 failures, a committed file, a posted GitHub review. Every skill must end with evidence, not claims. Agents grading their own work have positive bias — exit criteria keep them honest.
●
What is Anti-Rationalization?
LLMs will produce a plausible-sounding paragraph explaining why this particular task doesn't need a spec, or tests, or review. The table pre-empts these excuses. Example: "I'll write tests later" → "Later is the load-bearing word. There is no later."

When a document becomes a skill

My team-directory evolved through three stages. Click each to see the transformation — and the litmus test that separates a document from a skill.

team-directory.md
# Team Directory | Name | Slack ID | GitHub | |------------|---------------|------------------| | Venkat | U067BCU0GHE | venkateshwarans | | Anusha | U07QGKBFW4X | anusha-saras | | Waheed | U06V50U1308 | waheeddar05 | | Abhiram | U07UPNFRF6W | abhiram90 | | Jigar | U07M3P5E2SK | jigar1909 | | Pratishtha | U06UUSAL2NM | Pratishtha345 | | ...28 members total |
Just Data

A flat table of names and IDs. Useful to a human who knows what to do with it, but an agent has no instructions on what to do. It will grab a value and move on.

Checklist test: Can a human follow this as a step-by-step checklist? No. There are no steps. It's a spreadsheet.
team-directory/index.md
--- name: team-directory description: Maps Slack ↔ GitHub identities --- # Team Directory ## Overview A lookup table mapping 28 Saras team members between Slack User IDs, DM Channel IDs, and GitHub usernames. ## When to Use - Resolving a Slack user to GitHub username - Tagging the correct person on a PR - Sending a Slack message to a team member ## When NOT to Use - Looking up people outside Saras team - Finding email addresses (not stored) | Name | Slack ID | DM Channel | GitHub | |------------|-------------|-------------|--------------| | Venkat | U067BCU... | D066ND8... | venkatesh... | | ...
Better, But Still a Document

Now it has metadata, "When to Use," and structure. But it still has no steps, no exit criteria, no guardrails. It describes when to use the data, but not how. The agent reads it as context, not as a workflow.

Checklist test: Can a human follow this as a step-by-step checklist? Still no. "When to use" is guidance, not a process.
resolve-team-member/ ├── SKILL.md # Workflow: steps, exit criteria, guardrails ├── references/ │ └── directory.md # The team lookup table (data stays here) └── scripts/ # Optional: helper scripts
resolve-team-member/SKILL.md
--- name: resolve-team-member description: Resolve cross-platform identity and take action (tag, notify, assign) --- ## Steps 1. Identify the person from context (PR author, Slack sender, name mention) 2. Look up their row in references/directory.md 3. Determine the target platform GitHub? Slack? Both? 4. Execute the action: • GitHub → tag @username on PR • Slack → send DM via channel ID 5. Confirm action was taken ## Exit Criteria - Person tagged or notified on platform - Link to comment/message as evidence ## Anti-Rationalization "I'll just use the GitHub username" → Names don't always match. Always look up.
Now It's a Skill

The team table didn't disappear — it moved to references/directory.md. The data is still there, but now it's wrapped in a workflow. SKILL.md tells the agent how to use the data: identify, look up, decide, act, confirm.

Checklist test: Can a human follow this as a step-by-step checklist? Yes. Identify → look up → decide platform → execute → confirm. That's a workflow.

BIGGER THAN CLAUDE

Skills are an open standard

  • One skill, many agents Agent Skills is an open format by Anthropic, now adopted by 30+ tools. Write a skill once, use it in Claude Code, Cursor, Gemini CLI, Copilot, Kiro, and more.
  • A skill is just a folder with a SKILL.md Metadata + instructions + optional scripts. No proprietary plugin system. Version-controlled, portable, shareable.
  • Progressive disclosure keeps it lightweight Agents load only the name & description at startup. Full instructions load only when a task matches. Many skills, tiny context footprint.

YOUR TURN

Start with one skill

  • Pick your most repetitive workflow The process you explain the same way every time. Not a reference doc — a sequence of steps with a clear outcome.
  • Write steps, not essays Numbered steps. Exit criteria. Anti-rationalization table. Drop it in ~/.claude/skills/ and it's live.
  • Iterate from real usage Every time the agent takes a shortcut, add a rule. The skill gets smarter — like encoding your team's institutional knowledge.
The litmus test: If you could paste it into a PR checklist and a human could follow it step by step, it's a skill. If it reads like a style guide, it's reference material — and agents will skip it.
