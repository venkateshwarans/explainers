Agent Skills
Encoding Senior Engineering Discipline

AI agents take the shortest path to "done" — skipping specs, tests, and reviews. Skills are the harness component that enforces the invisible work.

continues from Part 1 workflow design SDLC phases
The Problem

Agents are junior engineers by default

When you ask an AI agent to implement a feature, it does exactly that. It writes the code, runs a quick check, and says "done." What it doesn't do — unless you tell it to — is write a spec first, create tests before the implementation, or prepare a proper code review.

This isn't a bug. It's the agent doing what it was optimized to do: reach "task complete" as efficiently as possible. That's the right behaviour for a language model. It's the wrong behaviour for a software engineer.

"The default behaviour of any AI coding agent is to take the shortest path to done." — Addy Osmani, Agent Skills

The invisible work of senior engineering — specification writing, test-driven development, scope discipline, code review — doesn't appear in code diffs. Nobody sees it. The agent has no signal telling it to do those things, so it skips them.

Skills are the harness component that fixes this. They're markdown workflow files that inject senior-engineering steps directly into the agent's execution path — with checkpoints that produce evidence, not just assertions.

What is a Skill, Exactly?

A skill is a workflow, not a document

A skill is a markdown file with frontmatter. But its purpose is not documentation — it's execution. The distinction matters enormously.

Reference material gets read once and ignored. A skill is a sequence of steps the agent follows, with checkpoints that produce concrete evidence, ending in a defined exit criterion. The agent doesn't summarize the skill — it follows it.

The test: If you could paste the skill into a PR checklist and a human could follow it step-by-step, it's a skill. If it reads like a style guide, it's reference material — and agents will skip it.

# /test — Verification Workflow

## Phase: Verify
Skills belong to one SDLC phase — they're loaded only when that phase is active

1. Read the spec in docs/spec.md before writing any test
2. Write a failing test first (red)
3. Implement the minimum code to make it pass (green)
4. Refactor without breaking the test (refactor)

## Exit criteria
Every skill must end with concrete evidence — not "I think it works"
✓ bun test output shows 0 failures
✓ Coverage includes the new code path
✓ No .skip() or .todo() in the test file

## Do not skip because:
Anti-rationalization table: pre-written counters to excuses the agent will generate
- "This is simple" → Acceptance criteria still apply. Five tests is fine. Zero is not.
- "Later" → There is no later. Tests written after shipping are rarely written at all.

The 6 SDLC Phases

Skills map to the full software development lifecycle

Addy's system organizes skills around six phases of software development. Each phase has a corresponding skill — a slash command the agent invokes when that phase is active. Click each phase to see what it checks.

Phase 1 · Define

Write the spec first

Before any code is written, this skill forces the agent to define what "done" looks like. What are the acceptance criteria? What exactly are you building? What are you not building?

Without a spec, agents drift. They implement what they think you want, which may not be what you need. Five lines of acceptance criteria prevents days of rework.

# /spec workflow

Steps:
1. Ask: what problem does this solve?
2. Define acceptance criteria (min 3)
3. List explicitly out-of-scope items
4. Write spec to docs/spec.md

Exit criteria:
✓ docs/spec.md exists and is >100 chars
✓ At least 3 acceptance criteria listed
✓ Out-of-scope section is non-empty
Phase 2 · Plan

Break work into vertical slices

Planning skill forces the agent to decompose the spec into concrete tasks — not "implement authentication" but "add password hashing to user model," "write JWT generation," "add login endpoint." Small, independently shippable units.

Agents that plan first make fewer wrong turns. They can verify each slice before moving on.

# /plan workflow

Steps:
1. Read docs/spec.md
2. Break into vertical slices (each deployable)
3. Order by dependency
4. Write tasks to docs/plan.md

Exit criteria:
✓ Every task maps to a spec criterion
✓ No task is >4 hours of estimated work
✓ Dependencies are explicit
Phase 3 · Build

Implement in scope-disciplined slices

The build skill enforces scope discipline — the agent touches only what it was asked to touch. No "while I'm here" refactors, no unsolicited improvements to adjacent files.

This prevents one of the most common agent failure patterns: a small feature turns into a major refactor that breaks unrelated things.

# /build workflow

Steps:
1. Read the current task from plan.md
2. Implement only that task
3. Do not touch files not in task scope
4. Mark task done in plan.md

Exit criteria:
✓ Only files in scope were modified
✓ Task marked complete in plan.md
✓ Existing tests still pass
Phase 4 · Verify

Prove it works — with evidence

Not "I ran the tests." The test skill requires concrete evidence: test output showing zero failures, coverage numbers, no skipped tests. The agent can't claim success without producing the artifact that proves it.

"Later is the load-bearing word. There is no later." Every feature ships with tests — full stop.

# /test workflow

Steps:
1. Write failing test first (TDD)
2. Implement to make it pass
3. Run full test suite
4. Save output to test-report.txt

Exit criteria:
✓ 0 test failures in output
✓ No .skip() in test files
✓ New code path has coverage
Phase 5 · Review

Catch what the builder missed

A separate review step — ideally a different agent or a fresh context — that reads the diff as an outsider. Did the implementation match the spec? Are there edge cases missing? Is there anything a future maintainer would find confusing?

Agents grading their own work have positive bias. The review skill breaks this by treating generation and evaluation as separate tasks.

# /review workflow

Steps:
1. Read spec.md, then read the diff
2. Check spec criteria are met
3. Check for missing edge cases
4. Write review.md with findings

Exit criteria:
✓ Each spec criterion checked (✓/✗)
✓ review.md is non-empty
✓ Reviewer is not the implementer
Phase 6 · Ship

Deploy safely — with a rollback plan

The ship skill is the last line of defense. Before anything goes to production, it checks: CI passes, docs are updated, a rollback procedure exists, and the team has been notified if the change is breaking.

Most agent disasters happen because nobody told the agent that "ship" means something specific in your system. This skill makes it explicit.

# /ship workflow

Steps:
1. Check CI is green
2. Check docs are updated
3. Document rollback procedure
4. Write deployment note

Exit criteria:
✓ CI passes on main
✓ CHANGELOG.md updated
✓ Rollback procedure documented
5 Design Principles

What makes a skill actually work

Not every markdown file works as a skill. These five principles separate skills that agents follow from reference material agents skip.

1
Process over prose
Steps and checklists drive action. Essays don't. A skill that says "ensure high code quality" will be ignored. A skill that says "run tsc --noEmit and save output to typecheck.txt" will be followed.
2
Anti-rationalization tables
Agents — like junior engineers — will generate plausible-sounding reasons to skip steps. "This is too simple for a spec." "Tests can wait." Pre-write the counter-argument directly in the skill file. The agent can't argue with a rule that already anticipated its excuse.
3
Verification is non-negotiable
Every skill must end with a concrete exit criterion — something the agent has to produce, not just assert. Not "tests pass." But "save the test output to test-report.txt showing 0 failures." Evidence beats claims.
4
Progressive disclosure
Don't load all 6 skills at startup. The harness reveals only the skill relevant to the current phase. A build session doesn't need the ship skill. Loading everything bloats context and confuses the agent about what phase it's in.
5
Scope discipline
"Touch only what you're asked to touch." Every skill should include this rule explicitly. The most common agent failure pattern is scope creep — one small change triggers an unasked-for refactor that breaks five other things.
Anti-Rationalization

Agents will try to skip steps. Skills have answers.

An agent optimizing for "task complete" will generate convincing-sounding reasons to skip anything that takes extra time. Skills pre-emptively answer every excuse. Here are the most common ones:

Demo · Common agent rationalizations and skill counter-arguments
What the agent says The skill's counter Phase
"This task is too simple to need a spec." Acceptance criteria still apply. Five lines is fine. Zero lines is not. /spec
"I'll write tests later, after the implementation." Later is the load-bearing word. There is no later. Tests are written before shipping — always. /test
"The tests pass, so it's ready to review." Tests passing is necessary but not sufficient. Review checks spec coverage, not just test coverage. /review
"While I'm in this file, I'll clean up these other things." Scope discipline: touch only what you're asked to touch. Open a new task for the cleanup. /build
"CI is green so it's ready to ship." CI green is table stakes. Docs updated? Rollback documented? Breaking change communicated? Ship skill checks all of these. /ship

Add your own rows as you discover new rationalizations in your specific codebase. Each row is a ratchet click — a failure encoded into the skill permanently.

How to Use Skills

Three ways to get started

You don't need to build all six skills at once. Start with the one phase where your agents most often take shortcuts. For most teams, that's either /spec (skipping requirements) or /test (shipping without tests).

Option 1 · Claude Code marketplace
Install Addy's skills directly from the Claude Code marketplace. Immediate access to all 6 SDLC skills, pre-tested and ready to use.
Option 2 · Write your own markdown files
Create .claude/skills/ in your project. Add markdown files following the format above. Works with Claude Code, Cursor, and any tool that supports markdown skill injection.
Option 3 · Adopt the patterns without tooling
Even without any agent tooling, the SDLC workflow patterns are valuable checklists for human engineers. Start by adding the exit criteria to your PR template — the habit forms before the automation.
Key insight: Skills connect directly to the ratchet principle from Part 1. Every time your agent skips a step, add a rule to the relevant skill — or add a new row to the anti-rationalization table. Over time, your skills encode your team's entire institutional knowledge about how software should be built.

Key Terms
Skill
A markdown workflow file that injects structured steps into an agent's execution. Not documentation — an actionable process with exit criteria.
Exit criterion
Concrete evidence the agent must produce before a phase is complete. Not "I think it works" — "test output shows 0 failures."
Anti-rationalization table
Pre-written rebuttals to the excuses agents will generate for skipping steps. Anticipated and answered in the skill file itself.
Progressive disclosure
Loading only the skill relevant to the current SDLC phase. Keeps context lean and tells the agent clearly what phase it's in.
Scope discipline
"Touch only what you're asked to touch." The rule that prevents agents from turning small changes into unasked-for refactors.
SDLC
Software Development Lifecycle. The 6 phases: Define, Plan, Build, Verify, Review, Ship. Each maps to one skill.
