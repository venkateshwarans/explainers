Long-Running Agents
Hours, Days, and Weeks

Most agents stop after one conversation. Long-running agents persist across context windows, sessions, and time — and the harness is what makes that possible.

context rot the Ralph loop brain/hands/session split
The Problem

What breaks when agents run for a long time

Imagine you ask an agent to build a full feature end-to-end. Not a one-liner — a real feature that involves reading the codebase, writing a spec, implementing across multiple files, running tests, and preparing a PR. That might take an hour or more of continuous operation.

Three things will go wrong, predictably, if the harness doesn't address them:

1
Context rot — the context window fills with old, irrelevant information. Quality degrades well before the hard limit is hit. The model starts making worse decisions around step 15 of a 40-step task.
2
No persistent state — if the session crashes, restarts, or hits a context limit, all progress is lost. The agent starts over with no memory of what it already tried.
3
No self-verification — models consistently over-report task completion accuracy. An agent that has been running for 45 minutes will say "done" even when it isn't, because it has no external check on its own optimism.

These aren't edge cases. They happen in every long-running agent. The harness has specific patterns to address each one.

Three Kinds of Long-Running

Not all long-running agents are the same

"Long-running" means three different things depending on context. The harness patterns you need differ for each. Click each type:

🧩
Long-horizon reasoning
Multi-step planning where each step depends on the previous one.
⏱️
Long-running execution
Processes spanning hours or days with thousands of model invocations.
🧬
Persistent agency
Agent identity that outlives individual tasks and accumulates memory over time.
Long-horizon reasoning: The agent plans 20-30 steps ahead and executes them with dependencies. The harness pattern is a plan file — a structured list of tasks the agent writes before starting. Each step is checked off. If the session resets, the plan survives in the filesystem.
Long-running execution: A coding sprint, a data analysis pipeline, or an automated testing run that takes hours. The harness pattern is checkpoint-resume — writing intermediate state to disk every N steps so a crash doesn't restart from zero.
Persistent agency: An agent that is "always on" — monitoring a system, triaging issues as they arrive, accumulating knowledge about a codebase over months. The harness pattern is a memory layer with identity controls, preventing "memory drift" as knowledge accumulates.
The Ralph Loop

The simplest pattern for long-horizon tasks

Here's a pattern that works for most individual developers doing multi-hour agent tasks. It was named after the person who first described it publicly. The key insight: state lives outside the agent's context. The filesystem maintains continuity. The model is stateless per-invocation.

Demo · The Ralph Loop — step through it
📋
Select incomplete task from tasks.md
The filesystem holds the task list. The agent reads it every loop iteration — not from memory, from disk. This is what survives context resets.
📝
Build prompt with task + persistent notes
The harness assembles the prompt fresh each loop: current task, notes from previous iterations, and the AGENTS.md rules. The model gets a clean, focused context — not the accumulated noise of 30 previous steps.
🤖
Invoke the agent
One focused invocation. The model works on one task with a clean context. Not "continue from where you left off" — a fresh call with all the context it needs, assembled by the harness.
✅
Run validation checks
External verification — not the agent checking its own work. Type check passes? Tests pass? Spec criteria met? These checks happen outside the model's context, preventing the self-grading bias.
📊
Append results to progress.log
Every iteration's results go to an append-only log. This is the session-as-event-log pattern — the most underappreciated piece of long-running agent design. If the process crashes, the log survives.
🔁
Update task status — repeat
Mark task complete in tasks.md. Select the next incomplete task. Loop. The harness intercepts any attempt to stop early and forces continuation until all tasks are complete.
Step 1 / 6

The key is that tasks.md and progress.log are the source of truth — not the model's context window. Any loop iteration can restart cleanly because state lives on disk.

Architecture

Brain, Hands, Session — Anthropic's decoupling

For production long-running agents, the harness decouples three concerns that most simple agents conflate:

Without the split
🤖 Model (everything mixed together)
reasoning, execution,
and state all in one context window
crashes lose all state
no recovery possible
With the Brain/Hands/Session split
🧠 Brain — model + harness loop
reasoning, planning, decision-making
🤝 Hands — ephemeral sandbox
code execution, file ops, network calls
📋 Session — append-only event log
every thought, tool call, observation
"The session-as-event-log idea is the part most teams underappreciate." — Addy Osmani, Long-Running Agents

The hands are ephemeral — they can crash, reset, or be swapped without losing any work. The brain focuses on reasoning, not execution details. The session log is the permanent record: if a container dies mid-task, the log survives. Recovery is possible because the log has everything.

The Evaluation Problem

Agents can't reliably grade their own work

Here's a subtle but critical problem: when an agent evaluates whether it's done, it has a positive bias. It just did the work. It wants it to be correct. This isn't a bug — it's an inherent feature of training on human feedback where correctness and effort correlate.

The solution is architectural: separate the roles. Cursor's approach uses three distinct roles:

🗺️
Planner
Explores codebase, emits tasks, spawns sub-planners. Never executes — only maps the work.
⚙️
Worker
Focused executor. No cross-coordination overhead. One task, one context, one verification step.
⚖️
Judge
Determines if iteration is complete. Triggers restarts when work doesn't meet criteria. Never the same agent that did the work.
Notable finding: Different model sizes excel at different roles. Frontier models (the most expensive/capable) sometimes underperform for extended autonomous work because they're more likely to stop prematurely. Smaller, faster models can outperform them as workers when the harness provides tight enough structure.
5 Production Patterns

How to build a long-running harness

Pattern 1
Checkpoint-and-resume
Write intermediate state to disk every N work units. When a container fails at step 23 of 40, the next invocation reads the checkpoint and continues from step 24. No rework, no data loss. Every long-running agent needs this.
Pattern 2
Delegated approval
Pause the agent mid-task with full execution state intact: reasoning chains, memory, tool history all preserved. A human reviews and approves. The agent resumes exactly where it stopped, with no context loss. This makes human-in-the-loop practical for long tasks.
Pattern 3
Memory-layered context
Govern accumulated knowledge like microservices. Different types of memory — short-term working memory, session memory, long-term project memory — have different read/write policies. Prevents "memory drift" where older context corrupts newer reasoning.
Pattern 4
Ambient processing
Event-driven agents that operate on streams without user interaction: monitoring CI results, triaging new issues as they're created, sending alerts when thresholds are crossed. The harness manages scheduling, not the model. Policy updates happen via gateway config, not code redeployment.
Pattern 5
Fleet orchestration
A coordinator agent spawns specialist sub-agents for isolated tasks. Each sub-agent has its own identity and context, preventing cascading failures — if one specialist fails, it doesn't corrupt the state of the others. The coordinator aggregates results.
Practical Guidance

Which approach to use when

Your situationRecommended approach
Individual dev, multi-hour tasks Ralph loop + plan.md + progress.log. Start here — simple and effective for 90% of cases.
Product with agents as a feature Managed runtime (Claude Managed Agents, Google Agent Platform). Don't build persistence from scratch.
Operational agents (monitoring, triaging) Managed memory + scheduled execution (Cloud Run + scheduler). Combine for tasks that accumulate state over invocations.
Multi-agent coordination Planner/worker/judge split. Coordinator handles delegation; specialists handle execution in isolation.
What to build first: Before context rot management, before fleet orchestration — invest in the session log. An append-only event log of every thought, tool call, and observation is the foundation everything else builds on. With a good session log, you can debug any failure. Without it, long-running agent failures are nearly impossible to diagnose.

Key Terms
Context rot
Degraded output quality as a model's context fills with stale information. Happens well before the hard context limit.
Ralph loop
A bash-script loop: select task → build prompt → invoke agent → validate → log → update status → repeat. State lives on disk, not in model context.
Brain/Hands/Session
Anthropic's decoupling: brain (reasoning), hands (execution sandbox), session (append-only event log). Each can fail independently.
Planner/Worker/Judge
Cursor's three-role model. Never let the implementer evaluate their own work.
Checkpoint-resume
Writing intermediate state every N steps. Enables recovery from failures without restarting from scratch.
Session log
Append-only record of every model invocation: thought, tool call, observation. The foundation of long-running agent recovery.
Memory drift
When accumulated agent memory from older sessions corrupts newer reasoning. Prevented by memory-layered context with identity controls.
Ambient processing
Event-driven agents operating on streams without user interaction. Policy via config, not code redeployment.
