What is a Harness?

How raw AI models get turned into agents that can actually do your work — and why the scaffolding matters more than the model.

no AI background required interactive demos build something at the end
Part 1

Start here: what is a harness, in everyday life?

Before we talk about AI, let's talk about harnesses you already know. A harness is a system of connections and constraints that lets something powerful do useful work safely.

Click each card to see how the idea maps to AI:

🧗
Climbing harness
Connects a climber to a rope. Doesn't limit how well they climb — limits how far they fall.
→ An AI harness connects the model to your tools and systems. It doesn't limit how smart the model is — it limits what happens when the model makes a mistake.
🐴
Horse harness
Channels a horse's strength into pulling a plow. Without it, the horse is powerful but useless for farming.
→ A raw AI model has capability, but no direction. The harness channels that capability into a specific task: review this PR, triage this issue, analyze this data.
⚡
Electrical harness
A bundle of wires that routes power and signals to the right place in a car. Without it: sparks everywhere.
→ An AI harness routes information: the right context to the model, tool results back to the right place, errors to the right handler. Without it: unpredictable behavior.
The pattern: A harness doesn't make the powerful thing more powerful. It makes the powerful thing useful, safe, and directed. That's exactly what a harness does for an AI model.
Part 2

A model is not an agent. What's missing?

Here's the thing most people misunderstand about AI: what you're paying for in a tool like Claude Code or GitHub Copilot is not the model. Models are increasingly commoditized. What you're paying for is the harness around it.

A raw AI model is just a text predictor. You give it text, it predicts what comes next. That's genuinely impressive — but it can't:

✗ Remember your codebase — it forgets everything after every conversation
✗ Run code — it can only generate text, not execute anything
✗ Enforce your rules — it might ignore your coding conventions without a system to enforce them
✗ Do multi-step tasks safely — it might delete the wrong file, expose a secret, or stop halfway through

The harness is what fills all these gaps. Let's build one, step by step.

Part 3

What goes wrong without a harness

Now you've seen what a harness is. Here's what happens in practice when teams skip it. These are real patterns you'll hit within days of building any serious AI feature.

Without harness
The agent has 40+ raw tools available all at once. It picks the wrong one, uses 3 tools when 1 would do, misses the right tool entirely, and exhausts its context window just processing all the options.
With harness
The harness uses progressive disclosure — loading only the tools relevant to the current task. The agent sees 5 focused tools instead of 40. It picks the right one every time.
Without harness
Every session starts cold. The agent asks you the same questions again. It doesn't know your team uses Bun, not npm. It doesn't know which directories are off-limits. It relearns everything every time.
With harness
The harness injects AGENTS.md into every session. Your conventions, past failures, and project context are always available. The agent starts every run knowing your codebase.
Without harness
The agent deletes files, commits to main, exposes API keys in source code, or overwrites generated files. It doesn't mean to — it's just doing what it was asked, without safety constraints.
With harness
Hooks catch destructive actions before they execute. The sandbox blocks unauthorized network calls. Secrets management keeps tokens out of model context entirely. Safety is structural, not trust-based.
Without harness
Every new AI feature needs custom session state, secret handling, output parsing, error recovery, and context management. Teams reinvent the same 400 lines of glue code for every agent.
With harness
A harness framework like Flue ships all of these as structured primitives. You configure, not reinvent. Focus goes to your domain logic — not the plumbing.
Part 4

The Ratchet: how a harness gets smarter over time

Here's one of the most powerful ideas in harness engineering. It's called the ratchet.

Every time your agent makes a mistake, you don't just fix it and move on. You encode a rule into the harness so that mistake can never happen again. The ratchet only clicks one way: your harness accumulates failures as permanent constraints.

This is completely different from how most people think about improving AI systems. You don't need a better model. You need a harness that's learned from every failure.

Part 6

Flue: a harness framework inspired by these ideas

Addy Osmani (Director of AI at Google Cloud) wrote a viral essay on X formalizing these ideas as "harness engineering." Fred K. Schott (co-creator of Astro) read an early draft and built Flue as a direct TypeScript implementation of the concepts.

Flue's core bet: the valuable part of an AI agent is the harness — so it should be something you own and can customize, not something you rent from a proprietary SDK.

Part 7 · Now go build

Your mental model, and what to do with it

You now understand what a harness is. Here's how to apply it:

Start here
Write your AGENTS.md
Create a file at the root of your project. Add your team's conventions: which package manager, which directories are off-limits, which patterns to always follow. Start small — 10 lines is enough. Every line should earn its place through a real situation.
When you hit a failure
Apply the ratchet
When your agent makes a mistake, ask: "how do I encode this into the harness so it never happens again?" Add a line to AGENTS.md, or a pre-commit hook, or a sandbox rule. Don't just retry — close the gap permanently.
Building a real agent
Try Flue
Flue gives you skills, sessions, sandbox, and secrets as structured TypeScript primitives. You focus on your domain logic — what the agent should actually do — rather than the plumbing around it.

Key Terms
Harness
The scaffolding around an AI model: prompts, tools, memory, sandbox, hooks. It makes a raw model into a useful, safe agent.
Agent
An AI model + a harness. The model handles language; the harness handles everything else.
AGENTS.md
A markdown file read by the agent every turn. Your project's conventions and rules — a checklist, not a style guide.
The Ratchet
The practice of encoding every agent failure as a permanent harness rule. Mistakes only get fixed forward — never retried and forgotten.
Sandbox
An isolated execution environment. The agent can run code inside it without risk to your actual system.
Hook
A script that runs at a lifecycle event (before a commit, after a file edit). Hooks enforce rules structurally, without relying on the model to behave well.
Progressive disclosure
Loading only the tools the agent needs for the current task, rather than all tools at startup. Prevents tool overwhelm and keeps context lean.
Context rot
Degraded output quality when a model's context fills with stale information during long tasks.
HaaS
Harness-as-a-Service. The shift to harness APIs that give you the full runtime — loop, tools, memory, sandbox — out of the box.
Flue
Open-source TypeScript harness framework. Built by Fred K. Schott after reading an early draft of Addy Osmani's essay on harness engineering.
