Why Flue

Everything you've learned — as code you own

Over the last three parts, you've learned that an agent is a model plus a harness, that skills enforce engineering discipline, and that long-running agents need persistent state and external verification. Flue is the framework that implements all of these ideas as structured TypeScript primitives.

The core premise: the harness is the valuable part. So it should be yours — fully owned, fully customizable, not rented from a proprietary SDK. Flue gives you the runtime. You bring the domain logic.

Other tools
Model + their harness. You configure what they allow. Hit a ceiling when your use case doesn't fit their assumptions.
Flue
Any model + your harness. You own every layer. Extend, fork, replace any component. No ceiling.
Before the Code

A quick decoder for the jargon ahead

The next section shows real Flue code. You'll see several library and platform names that may be unfamiliar. Here's what each one is, in plain terms:

TypeScript
JavaScript with type annotations. name: string means "this variable holds text." If you can read JavaScript, you can read TypeScript — the extra syntax is just labels.
FlueContext
An object Flue passes to your agent function. It contains init (to create a harness), payload (the input), and env (secrets/env vars).
valibot
A small library for validating data shapes. v.string() means "this must be text." Flue uses it to validate the agent's structured output.
just-bash
A lightweight virtual bash shell that runs in-process. Powers Flue's default sandbox — much faster than spinning up a real Docker container.
Daytona
A cloud service that gives you a real Linux container on demand. Use it when your agent needs a full development environment (git, Node, browser).
MCP
Model Context Protocol. A standard way for AI tools to plug into external services (GitHub, Slack, databases). Like USB for AI agents.
Durable Object
A Cloudflare Workers primitive that gives you stateful storage attached to a specific identifier. Flue uses these to persist session state on Cloudflare.
AGENTS.md
The markdown rulebook from Part 1. Flue reads it from your project root and injects it into every model call. Your team's conventions become the agent's defaults.
Don't memorize this. Hover or scroll back to this card as you read the code below. The point is to give you a mental footing — not to teach you every library upfront.
Core Concepts in Code

What a Flue agent looks like

Flue agents are TypeScript files. The harness concepts from this series map directly to code primitives. Click each tab to see a different layer:

// .flue/agents/review.ts — a PR review agent
import { type FlueContext } from '@flue/runtime';
import { local } from '@flue/runtime/node';
import * as v from 'valibot';

// Declares how this agent can be triggered
export const triggers = { webhook: true };

export default async function({ init, payload, env }: FlueContext) {
  // init() creates the harness — the control layer around the model
  // local() gives the agent access to the host filesystem + shell
  const harness = await init({
    model: 'anthropic/claude-sonnet-4-6',
    // Secrets stay in env — never exposed to model context
    sandbox: local({ env: { GH_TOKEN: env.GH_TOKEN } }),
  });

  // session() manages memory + conversation history
  const session = await harness.session();

  // skill() invokes a named markdown workflow — see next tab
  const { data } = await session.skill('review', {
    args: { prNumber: payload.prNumber },
    // Typed result validated with valibot schema
    result: v.object({
      verdict: v.picklist(['approve', 'request-changes']),
      summary: v.string(),
      issues: v.array(v.string()),
    }),
  });

  return data;
}

# /review — Code Review Workflow
# Saved to .agents/skills/review.md

## Phase: Review

## Steps
1. Run: gh pr view {args.prNumber} --json files,body
2. Read docs/spec.md if it exists
3. Read each changed file in the PR diff
4. Check each acceptance criterion from the spec
5. Run the test suite and record results
6. Populate the result schema with your verdict

## Exit criteria
✓ Every spec criterion marked approve or request-changes
✓ Test output recorded in issues[] if failures exist
✓ result.verdict is set based on findings

## Do not skip because:
- "Tests pass" → test coverage ≠ spec coverage
- "Small change" → smallest changes cause biggest bugs
// Sessions — memory across runs, built into harness.session()

// Same URL /{agent}/{id} continues the same session
const session = await harness.session();

// Named sessions — multiple threads inside one agent instance
const review = await harness.session('review-thread');
const research = await harness.session('research-thread');

// session.task() — a focused subagent in a detached session
// Shares the same sandbox/filesystem but gets fresh message history
const findings = await session.task(
  'Research the auth flow and list the key files.',
  { cwd: '/workspace/project', role: 'researcher' }
);

// Use the subagent's findings in the main session
await session.prompt(
  `Draft an implementation plan using:\n\n${findings.text}`
);

// On Cloudflare: session data backed by Durable Objects
// On Node.js: in-memory by default, provide custom store for persistence
// Three sandbox options — no custom config needed for most cases

// 1. Virtual sandbox (default) — fast, no container
//    Powered by just-bash, ideal for high-traffic agents
const harness = await init({ model: 'anthropic/claude-sonnet-4-6' });

// 2. Local sandbox — full host filesystem + shell (Node.js only)
//    Use in CI runners where isolation comes from the runner itself
import { local } from '@flue/runtime/node';
const harness = await init({
  sandbox: local({
    // Explicit env allowlist — only these vars reach the model
    env: { GH_TOKEN: process.env.GH_TOKEN },
  }),
  model: 'anthropic/claude-opus-4-7',
});

// 3. Remote sandbox (Daytona) — full Linux container
//    For coding agents that need git, Node.js, a browser
//    Install with: flue add daytona | claude
import { daytona } from '../connectors/daytona';
const sandbox = await client.create();
const harness = await init({ sandbox: daytona(sandbox) });
// MCP tools — connect external services without leaking secrets
import { connectMcpServer, type FlueContext } from '@flue/runtime';

export const triggers = { webhook: true };

export default async function({ init, payload, env }: FlueContext) {
  // Secret stays in env — model never sees the token
  const github = await connectMcpServer('github', {
    url: 'https://mcp.github.com/mcp',
    headers: { Authorization: `Bearer ${env.GITHUB_TOKEN}` },
  });

  try {
    const harness = await init({
      model: 'anthropic/claude-sonnet-4-6',
      // Pass MCP tools to the harness
      tools: github.tools,
    });
    const session = await harness.session();
    return await session.prompt(payload.prompt);
  } finally {
    await github.close();
  }
}
How it Works

How a request flows through Flue

When a user sends a message to your Flue agent, it passes through every layer of the harness before the model ever sees it — and again on the way out. Click each node to understand what that layer does:

Each box is a layer of your Flue harness. The model sits in the middle — surrounded by layers that give it context, capabilities, and constraints.

The model only sees what the harness assembles for it. Your system prompt, active skills, session context, and secrets are all injected by the harness — not by the user's message.

What Actually Happens

From flue dev to a running agent

Before the 5-step setup, here's what happens when you actually run a Flue agent. This is the mental model — what's going on behind the scenes when you fire up a development server.

# 1. You start the dev server
$ flue dev --target node
→ Flue scans .flue/agents/ for agent files
→ Starts HTTP server on port 3583
→ Watches files; reloads on change

# 2. You send a request
$ curl localhost:3583/agents/review/my-session \
    -d '{"prNumber": 42}'

# 3. Flue routes to your agent function
→ Loads .flue/agents/review.ts
→ Calls your default function({ init, payload, env })
  payload = { prNumber: 42 }

# 4. Your function calls init() — the harness boots
→ Sandbox starts (just-bash by default)
→ AGENTS.md loaded from project root
→ Skills indexed from .agents/skills/
→ Session opens (memory + history)

# 5. Your function calls session.skill('review', ...)
→ review.md loaded as workflow
→ Model called with skill instructions + payload
→ Model runs steps in workflow (tool calls in sandbox)
→ Returns structured result validated against schema

# 6. Your function returns the data
→ Flue serializes as JSON, sends HTTP 200 to client
→ Session persisted (for "my-session" id)

# Next request to same /my-session url = same session continues

Notice what you didn't have to write: the HTTP server, the routing, the sandbox setup, the model API client, the session persistence, the schema validation, the skill loading. Flue handles all of that. You write the orchestration: which skill to invoke, what payload, what shape of result to expect.

Getting Started

Your first Flue agent in 5 steps

1
Install Flue
bun add flue (or npm install flue). Flue is TypeScript-first — no separate type package needed.
2
Create your AGENTS.md
Start with your package manager, which directories are off-limits, and your team's critical conventions. 10 lines is a good first file. Every line should be earnable — add rules only when something goes wrong.
3
Define your first skill
Create .flue/skills/ and write one markdown workflow file. Start with the phase where your agent most often takes shortcuts — usually /spec or /test.
4
Configure sandbox + secrets
Define which commands are allowed. Move API keys out of your prompt and into the sandbox secrets layer. Even if the model makes a mistake, secrets stay hidden.
5
Deploy and observe
Use Flue's CLI (flue serve) for HTTP or flue run for CLI. Watch the session log. Every mistake the agent makes is your next AGENTS.md rule.

Flue Primitives
createAgent()
The top-level function. Takes model, systemPrompt, skills, sandbox, and session. Returns a deployable agent.
createSkills()
Loads markdown skill files from a directory. Each file maps to a slash command the agent can invoke.
createSession()
Sets up persistent state across runs. Configures what survives context resets and where it's stored.
createSandbox()
Configures the execution environment: allowed commands, network policy, and secrets injection.
createHooks()
Lifecycle enforcement: preToolUse, postEdit, preCommit. Blocks destructive actions structurally.
flue serve
Deploy as an HTTP server. One command — your agent becomes an API endpoint.
flue run
Run from CLI. For development, local scripts, and one-shot tasks.
