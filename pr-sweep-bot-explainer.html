How the PR Sweep Bot processes Slack events

TL;DR — A TypeScript/Bun process with three parallel detection paths for finding GitHub PR links in Slack messages. Any path that finds a new PR link fires a single action: spawn claude -p /pr-sweep as a subprocess. A Set<string> of Slack message timestamps prevents double-triggering across paths.

Architecture

Three paths converge to one action. The paths differ in latency and coverage — they're complementary, not redundant.

Detection paths

1 · Real-time events via Socket Mode index.ts:145–162

The Slack Bolt app connects over WebSocket (Socket Mode) and fires app.message for every message in channels the bot is invited to. It triggers immediately — no polling lag.

Two conditions must be true: the message is either a DM or mentions @Venkateshwaran, and it contains a GitHub PR link (or the word "review").

// index.ts:145
app.message(async ({ message }) => {
  if (message.subtype) return;           // skip edits, joins, etc
  const isDM      = msg.channel_type === "im";
  const mentionsMe = text.includes(`<@${MY_USER_ID}>`);

  if (!isDM && !mentionsMe) return;      // must be a DM or @-mention
  if (msg.user === MY_USER_ID) return;    // ignore own messages

  const prLinks = extractPRLinks(text);
  if (prLinks.length === 0 && !text.toLowerCase().includes("review")) return;
  if (seen.has(msg.ts)) return;           // already handled

  seen.add(msg.ts);
  setImmediate(() => runPRSweep(...));
});
// Socket Mode vs HTTP endpoints
//
// Traditional Slack bots need a public HTTPS endpoint for events.
// Socket Mode opens an outbound WebSocket instead — no inbound port,
// no ngrok, no cloud deploy needed.
//
// Trade-off: requires SLACK_APP_TOKEN (app-level token),
// and only receives events in channels the bot is invited to.
// That's why polling exists as a fallback for DMs.
2 · DM polling via conversations.history index.ts:51–98

Runs every 2 minutes using the user token (not the bot token) so it can read all DMs — even ones the bot isn't a member of. It calls conversations.list to enumerate DMs, then conversations.history on each to fetch messages since the last poll.

Why user token? The bot token only sees channels it's invited to. The user token reads as you — your DMs, your private channels. That's why there are two separate clients in the file.

// index.ts:51 — userClient reads as you
async function pollDMs() {
  const listRes = await userClient.conversations.list({
    types: "im,mpim",          // 1:1 DMs + group DMs
    exclude_archived: true,
    limit: 100,
  });

  for (const channel of listRes.channels) {
    const histRes = await userClient.conversations.history({
      channel: channel.id,
      oldest: String(lastPollTs),  // only fetch new messages
      limit: 20,
    });

    for (const msg of histRes.messages) {
      if (seen.has(msg.ts)) continue;
      const prLinks = extractPRLinks(msg.text);
      if (prLinks.length > 0) {
        seen.add(msg.ts);
        setImmediate(() => runPRSweep(...));
        return;                  // one trigger per poll
      }
    }
  }
}
// Why not use search.messages for DMs?
//
// search.messages is indexed asynchronously — new messages can take
// 5–10 minutes to appear in search results. That's the exact problem
// this bot was built to avoid (slow PR review turnaround).
//
// conversations.history is real-time: it reads the raw message store,
// not the search index. So DM polling uses history, not search.
//
// Channel messages use search.messages as a fallback only because
// conversations.history on a channel requires the bot to be a member.
3 · Channel search via search.messages index.ts:101–137

Covers public channels and channels the bot hasn't been invited to. Searches for github.com/pull and filters by recency. Results can lag 5–10 minutes behind real-time — it's the fallback path, not the primary.

DMs are explicitly skipped here (is_im || is_mpim) since pollDMs handles those faster.

// index.ts:101
async function searchChannels() {
  const res = await userClient.search.messages({
    query:    "github.com/pull",
    sort:     "timestamp",
    sort_dir: "desc",
    count: 20,
  });

  for (const msg of res.messages.matches) {
    if (seen.has(msg.ts)) continue;
    if (parseFloat(msg.ts) < oldest) break;  // past the window

    // Skip DMs — pollDMs() handles those with lower latency
    if (msg.channel?.is_im || msg.channel?.is_mpim) continue;
    if (msg.user === MY_USER_ID) continue;

    seen.add(msg.ts);
    setImmediate(() => runPRSweep(...));
    return;
  }
}

Deduplication

All three paths share one Set<string>. Slack message timestamps (ts) are globally unique — no two messages ever share one. The set grows in memory for the lifetime of the process and is never flushed, which is fine since the bot is typically restarted daily by launchd.

4 · The seen Set index.ts:19–21

A single check at each detection point prevents Socket Mode, DM polling, and channel search from all firing for the same message.

// index.ts:19
const seen = new Set<string>();   // ts → already triggered
let lastPollTs = Math.floor((Date.now() - POLL_INTERVAL_MS) / 1000);

// Every detection path does the same two lines before firing:
if (seen.has(msg.ts)) continue;   // skip if seen
seen.add(msg.ts);                    // mark before spawning
setImmediate(() => runPRSweep(context));
// Why an in-memory Set and not a DB or file?
//
// The bot is single-process and single-host (runs on your Mac via launchd).
// There's no concurrent writes to worry about — Node/Bun's event loop
// is single-threaded, so Set ops are atomically safe.
//
// A SQLite DB would survive restarts, but also risks re-triggering on
// old PRs after a reboot. The in-memory Set is intentionally ephemeral:
// each new session starts fresh, matching the poll window.

Spawning Claude

When any path fires, it calls runPRSweep(). A mutex flag (sweepRunning) ensures only one Claude process runs at a time — if a second trigger arrives while a sweep is in progress, it's dropped.

5 · Bun.spawn + /pr-sweep index.ts:28–47

Spawns claude -p /pr-sweep 30 minutes as a subprocess with the full environment inherited. stdout/stderr are piped to the parent process so logs appear in the same terminal window.

// index.ts:28
let sweepRunning = false;

function runPRSweep(context: string) {
  if (sweepRunning) return;   // one sweep at a time
  sweepRunning = true;

  const proc = Bun.spawn(
    [CLAUDE_PATH, "-p", "/pr-sweep 30 minutes"],
    { env: { ...process.env } as Record<string, string>,
      stdout: "inherit", stderr: "inherit" }
  );

  proc.exited.then(code => {
    sweepRunning = false;
    console.log(`[pr-sweep] Done (exit ${code})`);
  });
}
// Node.js child_process equivalent
import { spawn } from "child_process";

function runPRSweep(context: string) {
  if (sweepRunning) return;
  sweepRunning = true;

  const proc = spawn(
    CLAUDE_PATH,
    ["-p", "/pr-sweep 30 minutes"],
    { env: process.env, stdio: "inherit" }
  );

  proc.on("close", code => {
    sweepRunning = false;
    console.log(`[pr-sweep] Done (exit ${code})`);
  });
}

The key difference: Bun.spawn returns a promise-based proc.exited — no event listener boilerplate. Same semantics, half the lines.
